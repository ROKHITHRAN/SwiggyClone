================================================================================
SWIGGY CLONE - DATABASE MODELS RELATIONSHIPS & MOCK DATA GENERATION GUIDE
================================================================================

IMPORTANT NOTES FOR MOCK DATA GENERATION:
- Users must be created first (base entity)
- Then create specific user types (Customer, Restaurant, Delivery)
- Then create dependent entities in order of relationships
- Use consistent IDs for relationships with @MapsId relationships
- Enum values must match the defined enums exactly

================================================================================
1. ENUM VALUES REFERENCE
================================================================================

Role: CUSTOMER, RESTAURANT, DELIVERY_PARTNER

OrderStatus: PENDING, CONFIRMED, PREPARING, OUT_FOR_DELIVERY, DELIVERED, CANCELLED

FoodType: VEG, NON_VEG, BOTH

FoodCategory: APPETIZERS, MAINS, DESSERTS, BEVERAGES, BREADS, SOUPS, SALADS

RestaurantStatus: OPEN, CLOSED, MAINTENANCE

PaymentStatus: PENDING, SUCCESSFUL, FAILED, CANCELLED

PaymentMethod: CREDIT_CARD, DEBIT_CARD, UPI, WALLET, CASH_ON_DELIVERY

2. ENTITY DETAILS WITH REQUIRED FIELDS
================================================================================

─────────────────────────────────────────────────────────────────────────────
USERS (Base Entity - Must Create First)
─────────────────────────────────────────────────────────────────────────────
Primary Key: userId (Long - Auto Generated)

Fields:
  - userId: Long (Auto Generated)
  - userName: String (REQUIRED - unique identifier)
  - email: String (REQUIRED - unique)
  - phoneNumber: String (REQUIRED)
  - password: String (REQUIRED)
  - role: Role Enum (REQUIRED - CUSTOMER/RESTAURANT/DELIVERY_PARTNER)

Relationships:
  - ONE-TO-ONE: Address (one-sided, mappedBy with cascade)
  - ONE-TO-ONE: Images (one-sided, mappedBy with cascade)

Mock Data Requirements:
  - Minimum 1 User per role
  - Username must be unique
  - Email must be unique and valid format
  - Password should be encoded in real application
  - Phone number should follow valid pattern

Example Username Pattern: user_[role]_[number], e.g., user_customer_1, user_restaurant_1


─────────────────────────────────────────────────────────────────────────────
CUSTOMER
─────────────────────────────────────────────────────────────────────────────
Primary Key: userId (Long - Foreign Key to User)

Fields:
  - userId: Long (REQUIRED - @MapsId from User)
  - user: User (REQUIRED - OneToOne relationship)
  - isSubscribed: Boolean (REQUIRED - default false)
  - points: Integer (REQUIRED - default 0)
  - ratings: Integer (REQUIRED - default 0, range 0-5)

Relationships:
  - ONE-TO-ONE: User (Parent)
  - ONE-TO-ONE: Cart (one-sided, mappedBy with cascade)
  - ONE-TO-MANY: Orders (one-sided, mappedBy)
  - ONE-TO-MANY: Payment (via ManyToOne in Payment)

Dependency:
  - Must have corresponding User with role=CUSTOMER
  - userId must match User.userId

Mock Data Requirements:
  - isSubscribed: randomly true/false
  - points: 0-1000
  - ratings: 0-5


─────────────────────────────────────────────────────────────────────────────
RESTAURANT
─────────────────────────────────────────────────────────────────────────────
Primary Key: userId (Long - Foreign Key to User)

Fields:
  - userId: Long (REQUIRED - @MapsId from User)
  - user: User (REQUIRED - OneToOne relationship)
  - restName: String (REQUIRED)
  - ratings: Integer (REQUIRED - default 0, range 0-5)
  - description: String (OPTIONAL)
  - restaurantStatus: RestaurantStatus Enum (REQUIRED)
  - foodType: FoodType Enum (REQUIRED - VEG/NON_VEG/BOTH)

Relationships:
  - ONE-TO-ONE: User (Parent)
  - ONE-TO-MANY: FoodItems (one-sided, mappedBy with cascade & orphanRemoval)
  - ONE-TO-MANY: Orders (one-sided, mappedBy with cascade)

Dependency:
  - Must have corresponding User with role=RESTAURANT
  - userId must match User.userId

Mock Data Requirements:
  - restName: unique restaurant names
  - ratings: 0-5
  - description: restaurant description
  - restaurantStatus: OPEN or CLOSED
  - foodType: VEG, NON_VEG, or BOTH


─────────────────────────────────────────────────────────────────────────────
DELIVERY (Delivery Partner)
─────────────────────────────────────────────────────────────────────────────
Primary Key: userId (Long - Foreign Key to User)

Fields:
  - userId: Long (REQUIRED - @MapsId from User)
  - user: User (REQUIRED - OneToOne relationship)
  - currentLoc: Blob (REQUIRED - location data)
  - points: Integer (REQUIRED - default 0)

Relationships:
  - ONE-TO-ONE: User (Parent)
  - ONE-TO-MANY: Orders (one-sided, mappedBy)

Dependency:
  - Must have corresponding User with role=DELIVERY_PARTNER
  - userId must match User.userId

Mock Data Requirements:
  - currentLoc: latitude,longitude as Blob
  - points: 0-1000
  - ratings: 0-5

Relationships:

─────────────────────────────────────────────────────────────────────────────
CART
─────────────────────────────────────────────────────────────────────────────
Primary Key: cartId (Long - Auto Generated)

Fields:
  - cartId: Long (Auto Generated)
  - restaurant: Restaurant (REQUIRED - Foreign Key)
  - customer: Customer (REQUIRED - Foreign Key)
  - amount: Double (REQUIRED - total cart value)
  - cartItems: List<CartItems> (REQUIRED - cascade, orphanRemoval)

Relationships:
  - ONE-TO-ONE: Restaurant
  - ONE-TO-ONE: Customer (inverse relationship, mappedBy)
  - ONE-TO-MANY: CartItems (one-sided, cascade all, orphanRemoval)

Dependency:
  - Must have valid Restaurant and Customer references
  - Customer can have only ONE Cart at a time
  - All CartItems must belong to the same Restaurant as Cart

Mock Data Requirements:
  - One Cart per Customer
  - amount: calculated from CartItems (sum of food_cost * quantity)
  - restaurant: must correspond to Customer's selected restaurant

CARTITEMS

Fields:
  - cartItemId: Long (Auto Generated)

Relationships:
  - MANY-TO-ONE: FoodItems

  - FoodItems must exist and be available (isAvailable = true)
  - All CartItems in a Cart must reference same Restaurant

Mock Data Requirements:
  - quantity: 1-10
  - foodItems: from FoodItems table
  - cart: from Cart table
  - FoodItems.isAvailable must be true

================================================================================
3. CREATION ORDER FOR MOCK DATA
================================================================================

Follow this order to avoid foreign key constraint violations:

1. USER (Create at least):
   - 1 User with role=CUSTOMER
   - 1 User with role=RESTAURANT
   - 1 User with role=DELIVERY_PARTNER

2. CUSTOMER (Use Customer User)
   - Create Customer from User(role=CUSTOMER)

3. RESTAURANT (Use Restaurant User)
   - Create Restaurant from User(role=RESTAURANT)

4. DELIVERY (Use Delivery User)
   - Create Delivery from User(role=DELIVERY_PARTNER)

5. ADDRESS (Link to any User)
   - Create Address and link to User

6. IMAGES (Link to any User)
   - Create Images and link to User

7. FOODITEMS (Link to Restaurant)
   - Create FoodItems with valid Restaurant reference

8. CART (Link to Customer and Restaurant)
   - Create Cart with valid Customer and Restaurant

9. CARTITEMS (Link to Cart and FoodItems)
   - Create CartItems with valid Cart and FoodItems

10. ORDERS (Link to Customer, Restaurant, and optional Delivery)
    - Create Orders with valid Customer and Restaurant
    - Leave Delivery as null initially or assign later

11. ORDERITEMS (Link to Orders and FoodItems)
    - Create OrderItems with valid Orders and FoodItems (from same Restaurant)

12. PAYMENT (Link to Orders and Customer)
    - Create Payment with valid Orders and Customer

5. CONSTRAINTS & VALIDATIONS FOR MOCK DATA
================================================================================

UNIQUE CONSTRAINTS:
- User.email: Must be globally unique
- User.userName: Must be globally unique
- FoodItems.foodName per Restaurant: Unique within each restaurant

NULLABLE FIELDS:
- Orders.delivery: nullable (not assigned until delivery starts)
- FoodItems.descriptions: nullable
- FoodItems.imageUrl: nullable
- Address.user: can be null initially
- Images.user: can be null initially
- Delivery.orders: can be empty list

NON-NULLABLE FIELDS:
- All @Id fields (auto-generated)
- User: userName, email, phoneNumber, password, role
- Customer: user, isSubscribed, points, ratings
- Restaurant: user, restName, ratings, restaurantStatus, foodType
- Delivery: user, currentLoc, points, ratings
- FoodItems: foodName, ratings, cost, foodType, isAvailable, foodCategory, restaurant
- Cart: restaurant, customer, amount
- CartItems: foodItems, cart, quantity
- Orders: amount, orderStatus, restaurant, customer, payment
- OrderItems: foodItems, orders, quantity
- Payment: paymentStatus, paymentMethod, amount, customer

RANGE VALIDATIONS:
- *ratings: 0-5
- *quantity: > 0
- cost/amount: > 0
- points: >= 0
